1. Executive Summary (บทสรุปผู้บริหาร)

DEP-PM คือแพลตฟอร์มบริหารจัดการโปรเจกต์ (Project Management Platform) ที่ออกแบบให้ "มนุษย์และ AI Agent ทำงานอยู่บนบอร์ดเดียวกัน" ตั้งแต่ขั้นตอนวางแผน แตกงาน มอบหมายงาน ติดตามสถานะ ไปจนถึง Deploy ขึ้นระบบจริงแบบอัตโนมัติ โดยไม่แยกส่วนเครื่องมือ Planning ออกจากเครื่องมือ Execution เหมือนที่ทีมวิศวกรรมส่วนใหญ่เผชิญอยู่ (Jira วางแผน, GitHub Actions deploy, Slack คุยงาน กระจัดกระจายกันคนละที่)

แพลตฟอร์มนี้ต่อยอดจาก DEP v3.0 Engineering Platform โดยตรง — ใช้ Metadata Platform Engine เดิมสำหรับสแกนโปรเจกต์ที่มีอยู่แล้ว (Brownfield), ใช้ Documentation Engine และ AI Context Engine เดิมสำหรับให้ AI Agent เข้าใจบริบทของแต่ละโปรเจกต์ และใช้ Knowledge Graph เดิมสำหรับวิเคราะห์ผลกระทบก่อนมอบหมายงาน

จุดต่างที่ DEP-PM เพิ่มเข้ามาคือ Task Orchestration Layer ที่มอบหมายงานให้ AI Agent ทำจริง (ไม่ใช่แค่ให้ AI ช่วยเขียนเอกสาร) พร้อม Inter-Agent Communication Bus ที่ให้ Agent คุยงานส่งต่อบริบทกันเองได้ และ Automated Deployment Pipeline ที่ทริกเกอร์ทันทีเมื่องานผ่านการ review

สถาปัตยกรรมแบ่งเป็น 3 ชั้นหลัก: Intake Layer (รับโปรเจกต์ใหม่/เก่า) → Orchestration Layer (แตกงาน มอบหมาย สื่อสารระหว่าง Agent) → Delivery Layer (สถานะ + Deploy อัตโนมัติ) โดยชั้น Intake และ Context อ้างอิงกับ Engine ของ DEP v3.0 โดยตรงแทนการสร้างซ้ำ

4. Data Model (DB Schema)

ตารางหลักที่ต้องมีในรอบแรก (PostgreSQL) — ออกแบบตามรูปแบบ B2 ของ System Design Template โดยเพิ่มตารางเฉพาะของ PM/Agent ที่ DEP v3.0 ยังไม่มี

projects
  id (uuid, pk), name, type ('new'|'existing'), repo_url,
  status ('planning'|'active'|'paused'|'archived'),
  metadata_registry_ref (fk -> DEP Metadata Registry), created_at

tasks
  id (uuid, pk), project_id (fk), title, description,
  status ('backlog'|'planned'|'assigned'|'in_progress'|'review'|'done'|'deployed'),
  assignee_type ('human'|'agent'), assignee_id,
  agent_role ('pm'|'dev'|'senior_architect'), priority ('P0'|'P1'|'P2'|'P3'),
  depends_on (uuid[]), spec (text), estimate_points (int), created_at, updated_at

agents
  id (uuid, pk), name, role, provider ('anthropic'|'openai'|'google'),
  mode ('solo'|'team'), status ('idle'|'working'|'error'), last_active_at

agent_messages
  id (uuid, pk), project_id (fk), task_id (fk), from_agent_id, to_agent_id,
  message_type ('handoff'|'question'|'result'|'review_comment'),
  payload (jsonb), created_at

deployments
  id (uuid, pk), project_id (fk), task_id (fk), triggered_by ('auto'|'manual'),
  status ('queued'|'running'|'success'|'failed'), environment, commit_sha, created_at

audit_log
  id (uuid, pk), actor_type ('human'|'agent'), actor_id, action,
  entity_type, entity_id, diff (jsonb), created_at

5. Project Lifecycle (Task State Machine)

ทุก task ไม่ว่าจะมอบให้คนหรือ Agent ต้องไหลผ่าน state เดียวกันนี้เสมอ เพื่อให้ Dashboard และ Deploy Pipeline ทำงานร่วมกันได้แบบคาดเดาได้

stateDiagram-v2 [*] --> Backlog Backlog --> Planned: PM Agent breaks down requirement Planned --> Assigned: Orchestrator routes to Agent/Human Assigned --> InProgress: Agent/Human starts work InProgress --> Review: Output submitted Review --> InProgress: Revision requested (max 2 รอบ) Review --> Escalated: Revision fail 2 ครั้ง Escalated --> InProgress: Human/Senior Agent takes over Review --> Done: Approved Done --> Deployed: Auto-deploy triggered Deployed --> [*]

Escalation Rule ตรงกับ Max Revision = 2 ที่กำหนดไว้ใน AI Dev Team Guide เดิม — ถ้า Agent แก้ไม่ผ่าน 2 รอบ งานจะถูกยกระดับให้ Agent ที่สูงกว่า (Gemini SR) หรือมนุษย์เข้ามาดูแทนโดยอัตโนมัติ

6. New Project Onboarding Workflow

STEP 1

ผู้ใช้กรอก Requirement เป็นข้อความอิสระ (เหมือน preset ใน AI Dev Team Orchestrator) ผ่านหน้า "New Project"

STEP 2

Claude PM Agent วิเคราะห์และสร้าง Task Plan (JSON) พร้อม estimate และ dependency ตาม Template A2/A3 ของ Section Planning

STEP 3

ระบบสร้าง Project record + Task record ทั้งหมดใน status backlog แล้วแสดงให้ผู้ใช้ยืนยัน scope ก่อนเริ่มงานจริง

STEP 4

เมื่อยืนยัน — Orchestrator เปลี่ยนสถานะเป็น planned แล้วเริ่ม routing ไปยัง Agent/คนตาม Routing Rules (หัวข้อ 9)

7. Existing Project Onboarding (Brownfield)

สำหรับโปรเจกต์ที่มีอยู่แล้ว (dPRO AI Parking, MChat, Farm Lab หรือ repo ภายนอก) DEP-PM จะไม่เริ่มจากศูนย์ แต่เรียกใช้ DEP Metadata Platform Engine (DEP v3.0 หัวข้อ 5) สแกน Codebase ก่อน เพื่อให้ Task ที่สร้างขึ้นใหม่ "รู้จัก" ระบบเดิมจริงๆ ไม่ใช่แค่เดาจากชื่อไฟล์

STEP 1

ผู้ใช้เชื่อม Repository เข้าระบบ (Git URL / local path)

STEP 2

Metadata Platform Engine สแกนและสร้าง Metadata Record + อัปเดต Knowledge Graph ของโปรเจกต์นั้น

STEP 3

Gemini SR Agent วิเคราะห์ Metadata + Graph แล้วสร้าง "Baseline Report" — tech debt, missing tests, doc coverage ที่ขาด

STEP 4

Baseline Report ถูกแปลงเป็น Task เริ่มต้นใน Backlog โดยอัตโนมัติ ผู้ใช้เลือกได้ว่าจะทำ task ไหนก่อน หรือเพิ่ม requirement ใหม่ทับเข้าไป

ผลลัพธ์คือ Backlog ที่ผสมกันระหว่าง "หนี้เทคนิคที่ AI เจอเอง" กับ "งานใหม่ที่มนุษย์อยากทำ" อยู่บนบอร์ดเดียวกัน

8. Agent Orchestration Modes

ออกแบบเป็น 2 โหมดที่สลับได้ต่อโปรเจกต์ ไม่ผูกมัดกับ provider ใดตายตัว — เริ่มงานได้ทันทีด้วย Solo Mode แล้วอัปเกรดเป็น Team Mode เมื่อพร้อม

โหมดวิธีทำงานต้องมี API Keyเหมาะกับ
SOLO Claude ตัวเดียวสวมบทบาททั้ง PM, Developer และ Senior Reviewer ผ่าน system prompt ต่างชุดกันในแต่ละ task Anthropic เท่านั้น เริ่มต้นใช้งาน / โปรเจกต์เล็ก / ทีมที่ยังไม่มี budget หลาย provider
TEAM + + Claude PM + Codex Dev + Gemini SR ทำงานแยกจริงตาม SOW ที่กำหนดไว้ใน AI Dev Team Guide Anthropic + OpenAI + Google โปรเจกต์ที่ต้องการ code review อิสระจากคนละโมเดล / งาน production

กติกาสำคัญ: Task Orchestration Engine ไม่รู้ความต่างของโหมด — มันเห็นแค่ "Agent Role" (pm/dev/senior_architect) เท่านั้น ทำให้สลับโหมดได้โดยไม่ต้องแก้ logic การมอบหมายงานเลย เปลี่ยนแค่ config ว่า role ไหน map ไป provider ไหน

10. Inter-Agent Communication Protocol

Agent ทุกตัวสื่อสารผ่าน Message Bus เดียวกัน ไม่คุยกันตรงๆ นอกระบบ — ทุกข้อความถูกบันทึกลงตาราง agent_messages (หัวข้อ 4) เพื่อให้ Dashboard และ Audit Log เห็นบทสนทนาทั้งหมดแบบย้อนดูได้

ตัวอย่าง Message Schema (JSON) 
{
  "id": "msg_8841",
  "project_id": "proj_mchat",
  "task_id": "T003",
  "from_agent": "codex-dev",
  "to_agent": "gemini-sr",
  "message_type": "handoff",
  "payload": {
    "summary": "Implemented POST /v1/invoices endpoint",
    "files_changed": ["apps/mchat/api/invoice.py"],
    "request": "โปรด review security ก่อน merge"
  },
  "created_at": "2026-07-02T10:14:02Z"
}

ตัวอย่างการไหลของบทสนทนาจริงในหนึ่ง task:

PM → Orchestrator: "T003 พร้อม assign ให้ Codex — spec แนบแล้ว"
DEV → Gemini SR: "เขียน endpoint เสร็จแล้ว ขอ review security"
SR → Codex Dev: "พบ SQL injection เสี่ยงที่ query filter, แก้บรรทัด 42 แล้วส่งกลับ"
DEV → Gemini SR: "แก้แล้ว ส่ง revision รอบ 2"
SR → PM: "Approved — rating 9/10, พร้อม deploy"
PM → Deploy Pipeline: "T003 done, trigger auto-deploy"

12. Automated Deployment Workflow

เมื่อ task เข้าสถานะ done (ผ่าน review แล้ว) Orchestrator จะ trigger CI/CD ทันทีโดยไม่ต้องรอคนกดปุ่ม เว้นแต่โปรเจกต์นั้นตั้งค่า "Manual Approval Gate" ไว้

# .github/workflows/dep-pm-deploy.yml (ย่อ)
on:
  repository_dispatch:
    types: [task-done]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: dep-pm-cli verify --task ${{ github.event.client_payload.task_id }}
      - run: dep-pm-cli build
      - run: dep-pm-cli deploy --env staging
      - run: dep-pm-cli smoke-test --env staging
      - run: dep-pm-cli deploy --env production --if-approved
EnvironmentTriggerApproval
StagingTask เข้าสถานะ Done ทันทีไม่ต้อง — อัตโนมัติเต็มรูปแบบ
ProductionStaging smoke test ผ่านตาม config ต่อโปรเจกต์ — Manual gate ได้ถ้าต้องการ
