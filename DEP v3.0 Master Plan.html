1. Executive Summary (บทสรุปผู้บริหาร)

dPRO Engineering Platform (DEP) เวอร์ชัน 3.0 ได้ถูกกำหนดขึ้นเพื่อแก้ปัญหาคลาสสิกในอุตสาหกรรมการพัฒนาซอฟต์แวร์ นั่นคือ "การที่เอกสารระบบ สถาปัตยกรรม และบริบทการทำงานขาดการปรับปรุงให้สอดคล้องกับซอร์สโค้ดที่เกิดขึ้นจริง" ซึ่งก่อให้เกิดหนี้ทางเทคนิค (Technical Debt) และขัดขวางการนำระบบปัญญาประดิษฐ์ (AI Generative Tools) เข้ามาช่วยเหลือนักพัฒนาได้อย่างเต็มประสิทธิภาพ

วัตถุประสงค์หลักของ DEP v3.0 คือการปฏิวัติโครงสร้างวิศวกรรมให้เป็น Self-Documenting AI-Native Platform โดยสถาปัตยกรรมเทคโนโลยีทุกส่วนจะถูกควบคุมผ่าน Single Source of Truth ซึ่งก็คือการเปลี่ยนแปลงของ Codebase ระบบจะทำการดึงข้อมูล แปลงสภาพ และอัปเดตเอกสารสถาปัตยกรรม ผังการไหลข้อมูล API Docs แผนผังความรู้ (Knowledge Graph) ตลอดจนการตัดสินใจทางสถาปัตยกรรม (ADR) โดยอัตโนมัติ

การเปลี่ยนแปลงครั้งสำคัญในเวอร์ชัน 3.0 นี้ ถือเป็น "จุดเริ่มต้นและรากฐานที่สำคัญที่สุดสำหรับ เวอร์ชัน 4.0" โดยทุกเอกสารและโมดูลโครงสร้างจะถูกติดตั้งระบบฝัง Metadata, Cross-reference และแท็กความสัมพันธ์เชิงลึกที่เรียกว่า Semantic Anchor เพื่อพร้อมแปลงสภาพเข้าสู่ฐานข้อมูลความรู้ทางวิศวกรรม (Engineering Knowledge Base) และโมเดล RAG (Retrieval-Augmented Generation) ได้ทันทีในอนาคต ช่วยลดต้นทุนในการทำข้อมูลซ้ำซ้อน และเปลี่ยนองค์ความรู้ขององค์กรให้กลายเป็นระบบปัญญาประดิษฐ์อัจฉริยะ (Internal AI Assistant) ที่เข้าใจบริบททั้งในอดีต ปัจจุบัน และอนาคตอย่างสมบูรณ์แบบ

2. Vision & Mission (วิสัยทัศน์และพันธกิจ)

Vision (วิสัยทัศน์)

มุ่งมั่นสร้างมาตรฐานวิศวกรรมซอฟต์แวร์สายพันธุ์ใหม่ที่โค้ดและองค์ความรู้ทำงานหลอมรวมเป็นหนึ่งเดียว เพื่อให้ระบบนิเวศน์ทางเทคโนโลยีทั้งหมดขององค์กรสามารถพัฒนา ตรวจสอบ และอัปเดตตัวเองได้อย่างอัตโนมัติ (Autonomous Core) ปราศจากช่องว่างของข้อมูล และพร้อมรับการขยายตัวอย่างไร้ขีดจำกัด

Mission (พันธกิจ)

  • Unified Engine: สร้างแพลตฟอร์มวิศวกรรมกลางที่เป็นมาตรฐานร่วมสำหรับโปรเจกต์เชิงพาณิชย์หลัก ได้แก่ dPRO AI Parking (ระบบจัดการที่จอดรถอัจฉริยะ), MChat (แอปพลิเคชันบัญชีและการเงินขับเคลื่อนด้วยเสียงสำหรับผู้ประกอบการรายย่อย) และ Farm Lab (ระบบเกษตรกรรมแม่นยำและการผลิต Wolffia เชิงพาณิชย์) รวมถึงระบบใหม่ๆ ที่เกิดขึ้นในอนาคต
  • Eradicate Desynchronization: ขจัดปัญหาเอกสารไม่ตรงกับโค้ดจริง (Out-of-sync Documentation) โดยการบังคับใช้กลไก Auto-Generation จากระดับโค้ดขึ้นมาเป็นเอกสาร
  • AI-First Structuring: ออกแบบและควบคุมโครงสร้างเนื้อหาของระบบวิศวกรรมทั้งหมดให้อยู่ในรูปแบบที่ AI สามารถตีความและทำความเข้าใจความสัมพันธ์ได้ทันทีแบบ Machine-Readable

> **Status:** active | **Version:** 3.0.0 | **Domain:** shared

## 4. Related Standards & Compliance

มาตรฐาน Frontmatter ข้างต้นเป็นแกนหลักที่ทำให้ Metadata Platform Engine (หัวข้อถัดไป) สามารถดึงข้อมูลออกจากเอกสารทุกฉบับได้แบบอัตโนมัติ โดยไม่ต้องอาศัยการตีความเนื้อหาด้วย NLP เพียงอย่างเดียว ระบบ CI จะปฏิเสธ Pull Request ทันทีหากไฟล์ Markdown ในโฟลเดอร์ /docs ไม่มี Frontmatter ที่ถูกต้องตาม Schema ที่กำหนด

5. Metadata Platform Engine (แพลตฟอร์มสกัดเมทาดาทา)

Metadata Platform Engine คือหัวใจของ DEP v3.0 ทำหน้าที่อ่าน Codebase จริงของทั้ง 3 โปรเจกต์หลัก แล้วสกัด (Extract) โครงสร้างเชิงความหมายออกมาโดยอัตโนมัติผ่านเทคนิค AST (Abstract Syntax Tree) Parsing และ Static Code Analysis เฉพาะภาษา แทนที่จะพึ่งพาการเขียนเอกสารด้วยมือซึ่งมักจะ "หลุด sync" กับโค้ดจริงเมื่อเวลาผ่านไป

Source TypeExtraction MethodOutput Schemaใช้ในโปรเจกต์
Python (FastAPI / SQLAlchemy)ast module + docstring parserService / Endpoint SchemadPRO AI Parking, MChat
TypeScript / Reactts-morph AST traversalComponent Prop SchemaMChat, Platform Dashboard
SQL DDLsqlglot parserTable / Column / FK Schemaทั้ง 3 โปรเจกต์
IoT Config (YAML / MQTT Topic)Custom topic-tree parserDevice / Sensor SchemaFarm Lab

ผลลัพธ์จากทุก Extractor จะถูกรวมและ Normalize เข้าสู่ Unified Metadata Registry ซึ่งเป็นฐานข้อมูลกลาง (PostgreSQL + JSONB) — กติกาสำคัญคือ ห้ามแก้ไข Registry ด้วยมือโดยเด็ดขาด (Write-Only-By-Engine) เพื่อรับประกันว่าข้อมูลใน Registry สะท้อนสถานะจริงของ Codebase เสมอ

ตัวอย่าง Metadata Record ที่ถูกสกัดออกมาโดยอัตโนมัติ (JSON) 
{
  "entity_id": "svc-mchat-invoice-api",
  "source_file": "apps/mchat/api/invoice.py",
  "type": "fastapi_endpoint",
  "extracted_at": "2026-07-01T09:12:03Z",
  "signature": "POST /v1/invoices",
  "depends_on": ["db-mchat-invoices", "svc-mchat-auth"],

6. Documentation Engine

Documentation Engine อ่านข้อมูลจาก Metadata Registry แล้ว Render ออกมาเป็นเอกสาร 2 รูปแบบคู่ขนานกันเสมอ (Dual Output) เพื่อให้ตอบโจทย์ทั้งฝั่งมนุษย์และฝั่ง AI ไปพร้อมกัน — คุณสมบัติ "Self-Syncing" ที่ปรากฏใน Dashboard ด้านบนของเอกสารนี้มาจากกลไกของ Engine ตัวนี้โดยตรง

Output Formatวัตถุประสงค์Trigger การอัปเดต

ระบบ Templating ใช้โครงสร้างหัวข้อคงที่ (ตามมาตรฐานในหัวข้อที่ 4) ผสานเข้ากับ Component ที่ Reusable เช่น .kpi-card, .timeline และ mermaid diagram เพื่อให้เอกสารทุกฉบับในองค์กรมีหน้าตาและโครงสร้างการอ่านที่สอดคล้องกัน ลดภาระการทำความเข้าใจเอกสารใหม่ของทีมวิศวกรและของ AI Assistant เอง

7. AI Context Engine

เพื่อให้เอกสารทั้งหมดพร้อมใช้งานกับโมเดล AI ในลักษณะ Retrieval-Augmented Generation (RAG) ระบบจะทำ Modular Chunking โดยใช้ขอบเขตของหัวข้อระดับ H2 และ YAML Frontmatter เป็นตัวกำหนดขนาด Chunk (ประมาณ 512-800 tokens ต่อ chunk พร้อม overlap 10-15%) เพื่อรักษาบริบทให้ครบถ้วนเมื่อถูกดึงไปใช้แยกส่วน

graph LR MD[Markdown + Frontmatter] -->|Split by H2 + Overlap| Chunker[Chunking Service] Chunker -->|Embed| Embedding[Embedding Model] Embedding --> VectorStore[(AI Context Vector Store)] VectorStore -->|Similarity Search| Retriever[Context Retriever API] Retriever --> Assistant[V4.0 Internal AI Assistant] style VectorStore fill:#7c3aed,stroke:#fff,stroke-width:2px,color:#fff

เมื่อ Assistant ได้รับคำถาม ระบบจะไม่ได้ใช้ Similarity Search อย่างเดียว แต่จะนำผลลัพธ์ไป Re-rank ร่วมกับความสัมพันธ์ใน Knowledge Graph (หัวข้อที่ 9) เพื่อให้บริบทที่ตอบกลับมีทั้งความเกี่ยวข้องเชิงความหมายและความถูกต้องเชิงโครงสร้างของระบบจริง

ทุกครั้งที่ Agent ตัวใดตัวหนึ่งสร้างหรือแก้ไขเอกสาร/โค้ด ระบบจะบันทึก agent_id, timestamp และ prompt/spec ที่ใช้ ลงใน Audit Log (ดูหัวข้อที่ 12) เพื่อให้ทุกการเปลี่ยนแปลงสามารถตรวจสอบย้อนกลับได้เสมอ

11. Automation & DevOps

ทุกขั้นตอนตั้งแต่ commit จนถึง publish เอกสารถูกร้อยเข้ากับ CI/CD Pipeline เพื่อไม่ให้ Engineering Standards (หัวข้อที่ 4) เป็นเพียงแนวปฏิบัติที่ไม่มีการบังคับใช้จริง

STEP 1

Pre-commit Hook — ตรวจสอบ YAML Frontmatter Schema ของไฟล์ที่แก้ไขก่อนอนุญาตให้ commit

STEP 2

Pull Request — GitHub Actions รัน Metadata Diff และ Preview เอกสารที่จะถูก regenerate

STEP 3

Automated Review — Gemini SR Agent คอมเมนต์ผลตรวจ Architecture/Security ลงใน PR โดยอัตโนมัติ

STEP 4

Merge & Publish — Documentation Engine regenerate เอกสาร และ Knowledge Graph Sync ทันทีที่ merge เข้า main

# .github/workflows/dep-sync.yml (ย่อ)
on:
  push:
    branches: [main]
jobs:
  sync-metadata:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: dep-cli extract --scope changed
      - run: dep-cli docs:generate --dual-output
      - run: dep-cli graph:sync

12. Security & Compliance

เนื่องจาก MChat จัดการข้อมูลทางการเงินของผู้ประกอบการรายย่อย และ Farm Lab เก็บข้อมูลตำแหน่ง/เซนเซอร์ของฟาร์ม แพลตฟอร์มจึงต้องออกแบบให้สอดคล้องกับพระราชบัญญัติคุ้มครองข้อมูลส่วนบุคคล (PDPA) ควบคู่กับมาตรฐานความปลอดภัยของโค้ดเอง

ชั้นข้อมูล (Layer)ข้อกำหนด
Secrets & Credentialsห้าม hardcode ในโค้ดหรือเอกสารเด็ดขาด — ใช้ Environment Variable / Secret Manager เท่านั้น
Metadata Registry AccessRole-Based Access Control (RBAC) แยกสิทธิ์ Read/Write ตามทีมและ Service
Personal & Financial DataData classification tier ตาม PDPA พร้อม field-level encryption สำหรับข้อมูลอ่อนไหวของ MChat
Audit Trailบันทึกทุกการเปลี่ยนแปลงที่เกิดจาก AI Agent หรือมนุษย์ พร้อม timestamp และ actor

Security Audit ของทุก API Contract ใหม่จะถูกส่งให้ Gemini SR Agent ตรวจตาม OWASP Top 10 เป็นด่านแรก ก่อนที่จะเข้าสู่กระบวนการ Human Review ขั้นสุดท้าย

16. ADR Strategy

Architecture Decision Record (ADR) คือบันทึกการตัดสินใจเชิงสถาปัตยกรรมที่สำคัญ พร้อมเหตุผลและทางเลือกที่พิจารณา เพื่อให้ทีมในอนาคต (รวมถึง AI Assistant) เข้าใจ "ทำไม" ไม่ใช่แค่ "ทำอะไร"

Lifecycle

Proposed → Accepted → (Deprecated | Superseded) — ทุกสถานะถูกบันทึกไว้ใน Frontmatter ของไฟล์ ADR เอง และสะท้อนเข้า Knowledge Graph ทันที

ADR IDหัวข้อสถานะImplements/Used By
ADR-023Adoption of RESTful Envelope Pattern for Cross-Service CommunicationAcceptedDEP-STD-API-01
